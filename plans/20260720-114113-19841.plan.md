# Notes

## Objectives

- Add a full light theme alongside the existing dark theme for the single-page portfolio site.
- Provide a toggle control in the site header (`.statusbar`/`.sb-nav`) that switches the whole site's theme immediately, including the full-viewport WebGL hero canvas.
- Persist the user's explicit choice across visits (`localStorage`), defaulting to OS `prefers-color-scheme` when no explicit choice has been made.
- Avoid any flash of the wrong theme on initial load/reload, using Next.js 16's documented inline pre-hydration script pattern.
- Keep the change proportional: only color tokens change (neutrals get light equivalents; accent brand colors `--amber`/`--cyan`/`--green`/`--red` stay fixed); no redesign of layout, typography, or composition.

## Technical Specification

- **Functional Requirements:**
  - A light theme CSS variable set (new `--bg`, `--panel`, `--panel-solid`, `--line`, `--line-hi`, `--txt`, `--dim`, `--faint`, `--faint-txt` values), applied via a `data-theme="light"` attribute on `<html>`, overriding the existing dark `:root` defaults.
  - A visible, keyboard-focusable toggle button in `.sb-nav` (header), styled to match the existing `.f-links button` bracket convention (`[ dark ]` / `[ light ]`), with an `aria-label` describing the switch action relative to current mode.
  - Clicking the toggle updates `localStorage["theme"]`, sets `data-theme` on `<html>`, and immediately recolors every themed surface, including the Three.js scene's fog/background in `app/Effects.tsx`.
  - On first visit (no stored preference), the theme resolves from `matchMedia("(prefers-color-scheme: light)")`; once the user has explicitly toggled, the stored choice wins on all future visits regardless of subsequent OS-preference changes (per clarified-requirements assumption 4 â€” live-resync after an explicit choice is explicitly out of scope/optional and is not implemented here).
  - No flash of incorrect theme on initial paint, on any reload, in either OS-preference state.
  - All 11 non-definitional hardcoded `#04070d` literals in `app/globals.css` (excluding the `--bg: #04070d;` token definition itself) are replaced so they resolve through CSS variables/`color-mix()` instead, so no surface silently stays dark-only.
- **Non-Functional Requirements:**
  - No new npm dependencies (matches this codebase's existing precedent of hand-rolled client components â€” see `app/CopyEmail.tsx` â€” and avoids validating an unfamiliar third-party library's compatibility with this non-standard Next.js 16.2.10 build, per `AGENTS.md`).
  - No React Context is introduced; `app/Effects.tsx` and `app/ThemeToggle.tsx` both communicate purely through the DOM (`document.documentElement`'s `data-theme` attribute and `--bg` custom property), matching the "no Context exists yet" codebase context.
  - No test framework is configured (`package.json` has no test runner); verification for this plan is manual/visual. Automated coverage is a dependency for the Unit Test pipeline stage, not this plan.
  - Existing functionality (boot sequence, 3D tilt cards, count-up metrics, scroll reveals, copy-email button, `prefers-reduced-motion` handling) must continue to work unchanged in both themes.
- **Technical Architecture:**
  - Theme mechanism: `data-theme="dark" | "light"` attribute on `<html>` (`app/layout.tsx`), read/written by both a blocking inline `<script>` (anti-FOUC, first visit + reloads) and `app/ThemeToggle.tsx` (explicit user toggles). CSS selectors: `:root` (dark defaults, unchanged) and `:root[data-theme="light"]` (neutral overrides) in `app/globals.css`.
  - `app/Effects.tsx` observes the `data-theme` attribute via a `MutationObserver` (no Context, no custom events) and re-resolves `--bg` from `getComputedStyle` to update `scene.fog` whenever the theme changes, plus once at mount to match whatever theme the inline script already resolved before React hydrated.
  - Storage contract: `localStorage.getItem("theme")` / `.setItem("theme", "light" | "dark")`, shared verbatim between `app/layout.tsx`'s inline script and `app/ThemeToggle.tsx`.
- **Data Flow:**
  1. First visit / reload: browser parses `<head>` â†’ runs the inline script synchronously (before first paint) â†’ reads `localStorage["theme"]`, falls back to `matchMedia("(prefers-color-scheme: light)")` â†’ sets `data-theme` on `<html>` â†’ `:root[data-theme="light"]` CSS (if applicable) is already active before anything paints â†’ `app/Effects.tsx` mounts, resolves `--bg` via `getComputedStyle`, initializes `scene.fog` to match â†’ `app/ThemeToggle.tsx` mounts, reads the now-current `data-theme` attribute in a `useEffect`, and corrects its own label/`aria-label` a tick after first paint (imperceptible, since the page's actual colors were already correct pre-hydration).
  2. User clicks the toggle: `app/ThemeToggle.tsx` computes the opposite theme â†’ writes `localStorage["theme"]` â†’ sets `data-theme` on `<html>` â†’ browser-native CSS cascade recolors every token-driven surface instantly â†’ `app/Effects.tsx`'s `MutationObserver` fires â†’ recomputes `scene.fog` from the new `--bg` â†’ next animation frame renders the recolored WebGL scene â†’ the toggle's own React state updates its visible label.
  3. Return visit after an explicit toggle: the inline script now finds a valid stored value and uses it directly, ignoring system preference, reproducing flow (1) with the user's saved choice.
- **Implementation Strategy:**
  Three implementation approaches were evaluated for the theme mechanism:

  | Approach | Architectural alignment | Complexity | Risk | Maintainability | Total /20 |
  |---|---|---|---|---|---|
  | **A. `data-theme` attribute + inline pre-hydration `<script>` + CSS custom-property overrides, hand-rolled toggle** (chosen) | 5 | 4 | 4 | 5 | **18** |
  | B. React Context/`ThemeProvider` managing theme in JS state, consumed by toggle + Effects | 2 | 3 | 3 | 3 | 11 |
  | C. Adopt the `next-themes` npm package | 2 | 5 | 2 | 3 | 12 |

  - **A (chosen):** Matches `node_modules/next/dist/docs/01-app/02-guides/preventing-flash-before-hydration.md`'s own documented `## Themes` pattern for this exact Next.js 16.2.10 build (`data-theme` attribute set by a blocking inline script, `suppressHydrationWarning` on `<html>`, CSS attribute-selector overrides). It also matches this codebase's existing precedent of small, self-contained `"use client"` components with local `useState` (`app/CopyEmail.tsx`) and introduces zero new dependencies or abstractions (no Context exists anywhere in the app today). `app/Effects.tsx` can read theme state directly off the DOM (`MutationObserver` + `getComputedStyle`), so no provider/consumer wiring is needed at all.
  - **B (rejected):** A Context provider would still need the exact same inline anti-FOUC script underneath (Context alone cannot prevent a first-paint flash, since context values aren't known until React runs), so it adds an abstraction layer (provider, hook, consumer wiring) for zero functional benefit over A, and introduces a pattern (`Context`) that doesn't exist anywhere else in this single-page site.
  - **C (rejected):** `next-themes` would handle FOUC/system-preference/persistence with less hand-written code, but it's an unvalidated third-party dependency whose compatibility with this specific, non-standard Next.js 16.2.10 build is unverified (`AGENTS.md` explicitly warns that this Next.js version has breaking changes from training-data assumptions) and it contradicts this codebase's established "no state libraries, hand-roll small client components" precedent for what is ultimately a single boolean toggle.
- **Success Criteria:**
  - Site defaults to the OS `prefers-color-scheme` on first visit with no stored preference (light OS pref â†’ light theme; dark or no-preference â†’ dark theme).
  - A visible, keyboard-focusable toggle exists in `.statusbar`/`.sb-nav` with an accessible name indicating current mode and the switch action (e.g., `aria-label="Switch to light theme"` when currently dark).
  - Clicking the toggle switches all themed surfaces immediately: page background, panels, borders, text/dimmed/faint text, header bar, buttons, footer gradient, and the WebGL hero canvas's fog/background â€” with no remaining hardcoded-dark patch (the former `#04070d` literal usages).
  - The chosen theme persists across reloads and new tabs (not reverting to system default once explicitly chosen).
  - No flash of the wrong theme on initial load/reload in either OS-preference state.
  - Accent colors (amber/cyan/green/red) remain unchanged in both themes (out of scope to alter hex values); residual legibility against the light background is visually reviewed (see Risk Considerations).
  - Existing functionality (boot sequence, 3D tilt cards, count-up metrics, scroll reveals, copy-email button, `prefers-reduced-motion`) continues to work unchanged in both themes.
  - Toggle remains visible/reachable at mobile viewport widths (`max-width: 900px`), without regressing the existing mobile nav-link visibility rule.
- **Risk Considerations:**
  - **Amber-on-light contrast:** `--amber` (`#ffb224`) has a mid-range luminance, so its contrast ratio against *any* plausibly "light" background is inherently much lower than against the current dark background (a physical consequence of similar luminance, not fixable by `--bg` choice alone) â€” and accent hex values are explicitly out of scope to change. Mitigation: choose a moderately light (not stark-white) `--bg` for the light theme (`#dfe4ea`) to partially improve the ratio, and flag remaining amber-on-`--bg` legibility (e.g. `.h-eyebrow::before`, `.sec-head .idx`) for explicit visual sign-off in Plan Review/Code Review, per the clarified requirements' own deferral of exact contrast numbers to visual review rather than numeric spec.
  - **`color-mix()` CSS function browser support** (used to replace alpha-blended `#04070d88`/`#04070dcc`/`#04070dee` literals) requires a reasonably modern browser (Safari 16.4+/Chrome 111+/Firefox 113+). This is an acceptable floor given the site already relies on `backdrop-filter` throughout (`.statusbar`, `.mission`, `.btn`, etc.), which has a comparable support baseline.
  - **`THREE.Color` string parsing:** `app/Effects.tsx` resolves `--bg` via `getComputedStyle(...).getPropertyValue('--bg')`, which returns the literal authored CSS text (a `#rrggbb` hex string) for a custom property, not a computed `rgb()` value â€” this only holds as long as both `:root` blocks in `app/globals.css` keep `--bg` defined as a plain hex literal (true for both the existing dark value and the new light value proposed here).
  - **No automated test coverage today** â€” manual verification (multiple OS-preference states, explicit toggle, reload, new tab, mobile viewport) is the only verification path available at this stage; noted as a dependency for the Unit Test pipeline stage.
  - **`suppressHydrationWarning` on `<html>`** only affects hydration-mismatch diffing for that element's own attributes, not any descendant content, so its use here is narrowly scoped per Next.js's documented guidance and does not mask unrelated hydration issues elsewhere in the tree.
  - **Effects.tsx scene fidelity:** per the reuse-target boundary in the clarified requirements, only `scene.fog`/background tracking is made theme-aware; the cage wireframe (`0x1c2946`), orbit ring (`0x31456f`) and starfield (`0x3a4e75`) colors â€” which happen to match the *dark* theme's `--line`/`--line-hi` hues â€” intentionally remain hardcoded/unchanged in the light theme. This is a deliberate, in-scope-boundary simplification (matching "only color tokens change, not composition"), flagged here so reviewers don't mistake it for an oversight.
